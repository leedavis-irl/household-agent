**Author:** Claude (Engineer)

## Context

Iji is a household Chief of Staff running on EC2. The four calendar tools (query/create/modify/freebusy) are verified and working. The next three unbuilt High-priority items in Scheduling & Coordination share a common dependency: a scheduler that can fire actions at specific times.

- **Reminders** need per-minute checks to fire DMs at the right time
- **Morning briefing** needs a once-daily trigger that gathers context and sends a summary

Both use the same infrastructure: a polling loop, a SQLite table of scheduled jobs, and a Signal delivery path. Building them together avoids duplicate plumbing.

Read ARCHITECTURE.md for the four-flow design and tool patterns.

## Goal

1. Build a shared scheduler that checks for due jobs every 60 seconds
2. Give household members the ability to set, list, and cancel reminders
3. Deliver a proactive morning briefing to Lee (and eventually all adults) via Signal

## User Experience

### Reminders

**Setting:**
- "Remind me to call the dentist tomorrow at 10am"
- "Remind Steve to take out the trash at 6pm"
- "Remind me in 30 minutes to check the oven"

**Listing:**
- "What reminders do I have?"
- "Show all pending reminders"

**Canceling:**
- "Cancel reminder #3"
- "Cancel my reminder about the dentist"

**Firing:**
- At the scheduled time, Iji sends a Signal DM: "⏰ Reminder: Call the dentist"
- Cross-person reminders include the requester: "⏰ Reminder from Lee: Take out the trash"

### Morning Briefing

Every day at 7:00 AM Pacific, Iji sends Lee a Signal DM with a summary. The content is generated by Claude — Iji calls its own brain with a "generate morning briefing" prompt that has access to all tools. Claude decides what's worth mentioning.

**Typical briefing might include:**
- Today's calendar highlights and conflicts
- Weather conditions and outfit guidance
- Pending reminders firing today
- Anything stored in knowledge base yesterday

**What it does NOT include:** Iji doesn't pre-fetch everything and stuff it into the prompt. It calls the brain with a briefing instruction, and Claude uses tools (calendar_query, weather_query, knowledge_search, reminder_list) to gather what it needs. This follows the "tools, not pre-fetch" principle from ARCHITECTURE.md.

## What to Build

### Component 1: Reminders SQLite Table

**File:** `src/utils/db.js` (Modify — add to migrate function)

```sql
CREATE TABLE IF NOT EXISTS reminders (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  content TEXT NOT NULL,
  target_person_id TEXT NOT NULL,
  requested_by TEXT NOT NULL,
  fire_at TEXT NOT NULL,
  created_at TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  fired_at TEXT
);
CREATE INDEX IF NOT EXISTS idx_reminders_fire_at ON reminders(fire_at);
CREATE INDEX IF NOT EXISTS idx_reminders_status ON reminders(status);
```

Column notes:
- `target_person_id`: who receives the reminder (household member id, e.g. 'lee', 'steve')
- `requested_by`: who asked for it (may differ from target)
- `fire_at`: ISO 8601 timestamp in UTC
- `status`: 'pending', 'fired', 'cancelled'
- No `channel` column — delivery is always Signal for now. Add column later if needed.

### Component 2: `reminder_set` Tool

**File:** `src/tools/reminder-set.js` (Create)

**Tool definition name:** `reminder_set`

**Description:** Set a reminder for a household member. Iji will deliver the reminder via Signal DM at the specified time.

**Input schema:**
- `person` (string, optional): Who receives the reminder. Member id or display name. Default: the person speaking.
- `content` (string, required): What to remind them about.
- `fire_at` (string, required): When to fire. ISO 8601 UTC timestamp. Claude converts relative times ("in 30 minutes", "tomorrow at 10am") to absolute UTC. Household timezone is America/Los_Angeles.

**Execution logic:**
1. Resolve `person` to a member id. Default to `envelope.person_id` if omitted. Use the same `resolvePersonId()` pattern from email tools.
2. Validate target person exists and has a Signal number in `identifiers.signal`
3. Validate `fire_at` is a parseable date in the future
4. Insert into reminders table: content, target_person_id, requested_by (from envelope.person_id), fire_at, created_at=now, status='pending'
5. Return: `{ reminder_id: row.id, content, target: display_name, fire_at_local }` where `fire_at_local` is fire_at formatted in Pacific time for the confirmation message ("I'll remind you at 10:00 AM Pacific")

**Error cases:**
- Unknown person → "Could not identify who to remind. Use a member name like 'steve' or 'lee'."
- No Signal number → "[Name] doesn't have Signal configured, so I can't deliver the reminder."
- Past time → "That time is in the past. Please specify a future time."

### Component 3: `reminder_list` Tool

**File:** `src/tools/reminder-list.js` (Create)

**Tool definition name:** `reminder_list`

**Description:** List pending reminders for a household member.

**Input schema:**
- `person` (string, optional): Whose reminders to list. Default: the person speaking.
- `include_fired` (boolean, optional): Include reminders that already fired today. Default: false.

**Execution logic:**
1. Resolve person id (default to envelope.person_id)
2. Query reminders where target_person_id = person and status = 'pending'. If include_fired, also include status='fired' where fired_at is today.
3. Order by fire_at ascending
4. Format each as: `#[id] — [content] (fires at [time Pacific])` or `#[id] — [content] (fired at [time Pacific] ✓)`
5. Return: `{ reminders: [...], message: "You have N pending reminders." }`

### Component 4: `reminder_cancel` Tool

**File:** `src/tools/reminder-cancel.js` (Create)

**Tool definition name:** `reminder_cancel`

**Description:** Cancel a pending reminder by ID.

**Input schema:**
- `reminder_id` (number, required): The reminder ID from reminder_list results.

**Execution logic:**
1. Look up reminder by id
2. If not found or already fired/cancelled → error message
3. Verify the person speaking is the target, the requester, or has admin role
4. Update status to 'cancelled'
5. Return: `{ cancelled: true, content: reminder.content }`

### Component 5: Reminder Scheduler

**File:** `src/utils/reminder-scheduler.js` (Create)

A polling loop that runs every 60 seconds. Follow the pattern from `usage-log.js` — export a `startReminderScheduler()` function.

**Every 60 seconds:**
1. Query: `SELECT * FROM reminders WHERE status = 'pending' AND fire_at <= ?` with current UTC ISO timestamp
2. For each due reminder:
   a. Look up target person from household config via `getHousehold()`
   b. Get their Signal number from `identifiers.signal`
   c. Format message:
      - Self-reminder: `⏰ Reminder: [content]`
      - Cross-person: `⏰ Reminder from [Requester Name]: [content]`
   d. Call `sendMessage(signalNumber, formattedMessage)` — imported from `src/broker/signal.js`
   e. If `sendMessage()` returns false → log error, leave as pending (retry next cycle)
   f. If returns true → update status='fired', fired_at=now()
3. Log: `Reminders checked: N due, M fired, K failed`

**Cleanup (once daily, checked each cycle):**
- Delete reminders where status='fired' AND fired_at < 7 days ago
- Delete reminders where status='cancelled' AND created_at < 1 day ago
- Track last cleanup date in a module-level variable (same pattern as `lastDailySummaryDate` in usage-log.js)

### Component 6: Morning Briefing Scheduler

**File:** `src/utils/morning-briefing.js` (Create)

A once-daily trigger. Checked every 60 seconds as part of the same interval or its own.

**Logic:**
1. Every 60 seconds, check: is it between 7:00 AM and 7:01 AM Pacific, AND has the briefing NOT been sent today?
2. If yes:
   a. Build a synthetic envelope for Lee:
      ```js
      {
        person_id: 'lee',
        person: 'Lee',
        role: 'admin',
        permissions: [...lee's permissions from household config...],
        message: 'Generate my morning briefing for today.',
        source_channel: 'signal',
        reply_address: lee's signal number,
        conversation_id: `briefing-${todayDateString}`,
        timestamp: new Date().toISOString()
      }
      ```
   b. Call `think(envelope)` from `src/brain/index.js` — this invokes Claude with full tool access. Claude will use calendar_query, weather_query, reminder_list, knowledge_search as it sees fit.
   c. Send the response to Lee via Signal DM using `sendMessage()`
   d. Mark today's briefing as sent (module-level variable)
3. On failure, log the error. Do not retry until next day — a stuck briefing loop would be expensive (each briefing is a full Claude API call with potential tool use).

**Briefing prompt:** The synthetic message should be a guided instruction, not just "generate a briefing." Use this message:

```
Generate a morning briefing for [Person Name]. Today is [date].

Check the following and include anything noteworthy:
1. Their calendar for today — events, times, locations. Also check for conflicts across household members (e.g., two adults both marked for school pickup but one has a conflicting meeting).
2. Current weather conditions and today's forecast — mention if it affects plans (rain during outdoor event, cold snap, etc.).
3. Their pending reminders firing today.
4. Anything stored in household knowledge in the last 24 hours that's relevant to them.

Keep it concise — this is a Signal message, not an email. Lead with the most important item. Skip sections that have nothing noteworthy (don't say "no reminders today" — just omit it). Write naturally, like a Chief of Staff giving a verbal briefing.
```

This guided prompt ensures consistency while letting Claude compose freely. As we add more tools (email, finance, Trello), we can extend the numbered list without changing briefing infrastructure.

**Configuration:**
- `BRIEFING_HOUR=7` — env var, defaults to 7 (AM Pacific)
- `BRIEFING_RECIPIENTS=lee,kelly` — env var, comma-separated person ids. Defaults to 'lee,kelly'. Expand to other adults later via opt-in/out (see backlog).
- `BRIEFING_ENABLED=true` — env var, defaults to true. Allows disabling without code change.
